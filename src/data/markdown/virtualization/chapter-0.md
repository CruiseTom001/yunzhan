## 1.1 什么是虚拟化

虚拟化（Virtualization）是将物理计算资源抽象为虚拟资源的技术，使得多个操作系统或应用可以共享同一物理硬件。

### 虚拟化的核心价值

| 价值 | 说明 |
|------|------|
| 资源整合 | 多个应用共享一台物理机，提高利用率 |
| 隔离性 | 虚拟机之间相互隔离，互不影响 |
| 灵活性 | 快速创建、迁移、克隆虚拟机 |
| 成本降低 | 减少物理服务器数量，降低硬件和运维成本 |
| 高可用 | 虚拟机可在线迁移，实现不停机维护 |
| 快照备份 | 快速创建和恢复快照 |

### 虚拟化前 vs 虚拟化后

\`\`\`
虚拟化前（一应用一服务器）：
┌──────────┐  ┌──────────┐  ┌──────────┐
│  App A    │  │  App B    │  │  App C    │  CPU 利用率 5-15%
│  OS       │  │  OS       │  │  OS       │
│  硬件     │  │  硬件     │  │  硬件     │
└──────────┘  └──────────┘  └──────────┘

虚拟化后（多应用共享服务器）：
┌─────────────────────────────────────────┐
│  App A  │  App B  │  App C  │  App D    │  CPU 利用率 60-80%
│  VM 1   │  VM 2   │  VM 3   │  VM 4     │
│─────────────────────────────────────────│
│            Hypervisor (VMM)              │
│─────────────────────────────────────────│
│              物理硬件                     │
└─────────────────────────────────────────┘
\`\`\`

## 1.2 虚拟化类型

### 按虚拟化层次分类

| 类型 | 原理 | 代表 | 性能 | 兼容性 |
|------|------|------|------|--------|
| 全虚拟化 | 完全模拟硬件，Guest OS 无需修改 | VMware Workstation, KVM (qemu) | 较好 | 好（支持未修改的 OS） |
| 半虚拟化 | Guest OS 需要修改以配合虚拟化 | Xen (PV 模式) | 好 | 差（需要修改 Guest OS） |
| 硬件辅助虚拟化 | CPU 提供虚拟化指令集支持 | KVM + VT-x/AMD-V | 最好 | 好（需要 CPU 支持） |
| 容器虚拟化 | OS 级别隔离，共享内核 | Docker, LXC | 最好 | 限同类型 OS |

### 全虚拟化（Full Virtualization）

\`\`\`
┌──────────┐  ┌──────────┐
│  App A    │  │  App B    │
│  Guest OS │  │  Guest OS │  ← 无需修改的操作系统
├──────────┤  ├──────────┤
│ 二进制翻译 │  │ 二进制翻译 │  ← 拦截特权指令并模拟
├─────────────────────────────┤
│        Hypervisor           │
├─────────────────────────────┤
│          硬件               │
\`\`\`

- Guest OS 完全不知道自己运行在虚拟环境中
- Hypervisor 需要拦截所有特权指令并模拟执行
- 性能有一定损耗（二进制翻译开销）

### 半虚拟化（Paravirtualization）

\`\`\`
┌──────────┐  ┌──────────┐
│  App A    │  │  App B    │
│  修改的OS │  │  修改的OS │  ← 内核被修改，主动调用 Hypercall
├──────────┤  ├──────────┤
│ Hypercall │  │ Hypercall │  ← 直接与 Hypervisor 通信
├─────────────────────────────┤
│        Hypervisor           │
├─────────────────────────────┤
│          硬件               │
\`\`\`

- Guest OS 知道自己运行在虚拟环境中
- 通过 Hypercall 直接与 Hypervisor 通信
- 性能比全虚拟化好（无需二进制翻译）
- 缺点：需要修改 Guest OS 内核

### 硬件辅助虚拟化

\`\`\`
┌──────────┐  ┌──────────┐
│  App A    │  │  App B    │
│  Guest OS │  │  Guest OS │  ← 无需修改
├──────────┤  ├──────────┤
│  VT-x /   │  │  VT-x /   │  ← CPU 硬件级虚拟化支持
│  AMD-V    │  │  AMD-V    │
├─────────────────────────────┤
│        Hypervisor           │
├─────────────────────────────┤
│          硬件               │
\`\`\`

- Intel VT-x / AMD-V 提供 CPU 虚拟化指令
- VT-d / IOMMU 提供 I/O 设备直接访问
- 几乎没有性能损耗
- 现代虚拟化方案的基础

\`\`\`bash
# 检查 CPU 是否支持硬件虚拟化
grep -E '(vmx|svm)' /proc/cpuinfo
# vmx = Intel VT-x
# svm = AMD-V

# 检查 KVM 模块是否加载
lsmod | grep kvm
\`\`\`

### 容器虚拟化

\`\`\`
┌──────────┐  ┌──────────┐  ┌──────────┐
│  App A    │  │  App B    │  │  App C    │
│  Libs     │  │  Libs     │  │  Libs     │
├──────────┤  ├──────────┤  ├──────────┤
│ Container│  │ Container│  │ Container│  ← 进程级隔离
├──────────┴──┴──────────┴──┴──────────┤
│          宿主机内核 (Host Kernel)       │  ← 共享内核
├───────────────────────────────────────┤
│              硬件                      │
\`\`\`

- 不模拟硬件，只隔离进程
- 共享宿主机内核，启动快、开销小
- 隔离性弱于虚拟机

## 1.3 Hypervisor 分类

| 类型 | 说明 | 代表 |
|------|------|------|
| Type 1（裸金属） | 直接运行在硬件上 | VMware ESXi, KVM, Xen, Hyper-V |
| Type 2（托管型） | 运行在宿主操作系统上 | VMware Workstation, VirtualBox, QEMU |

\`\`\`
Type 1:                    Type 2:
┌────┐ ┌────┐             ┌────┐ ┌────┐
│VM 1│ │VM 2│             │VM 1│ │VM 2│
├─────┴─────┤             ├─────┴─────┤
│ Hypervisor│             │ Hypervisor│
├───────────┤             ├───────────┤
│   硬件    │             │  宿主OS   │
└───────────┘             ├───────────┤
                          │   硬件    │
                          └───────────┘
\`\`\`

## 1.4 虚拟化的关键概念

| 概念 | 说明 |
|------|------|
| Guest OS | 运行在虚拟机中的操作系统 |
| Host OS | 运行 Hypervisor 的宿主操作系统（Type 2） |
| vCPU | 虚拟 CPU，从物理 CPU 中分配的时间片 |
| vNIC | 虚拟网卡 |
| vDisk | 虚拟磁盘（文件形式存储） |
| Snapshot | 虚拟机快照，保存某一时刻的状态 |
| Migration | 虚拟机从一台物理机迁移到另一台 |
| Overcommit | 超额分配（分配的 vCPU/内存超过物理总量） |
