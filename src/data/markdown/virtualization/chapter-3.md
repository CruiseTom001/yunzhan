## 4.1 原理对比

### 虚拟机架构

\`\`\`
┌──────┐ ┌──────┐ ┌──────┐
│ App  │ │ App  │ │ App  │
│ Libs │ │ Libs │ │ Libs │
│Guest │ │Guest │ │Guest │   ← 每个虚拟机有完整的 Guest OS
│ OS   │ │ OS   │ │ OS   │
├──────┤ ├──────┤ ├──────┤
│Hypervisor（硬件虚拟化）│
├───────────────────────┤
│    宿主机 OS / 硬件    │
\`\`\`

### 容器架构

\`\`\`
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│ App  │ │ App  │ │ App  │ │ App  │
│ Libs │ │ Libs │ │ Libs │ │ Libs │
├──────┤ ├──────┤ ├──────┤ ├──────┤
│Container│Container│Container│Container│  ← 进程级隔离
├───────────────────────────────────────┤
│         容器运行时（Docker）            │
├───────────────────────────────────────┤
│           宿主机内核                    │   ← 共享内核
├───────────────────────────────────────┤
│             硬件                       │
\`\`\`

### 核心差异

| 特性 | 虚拟机 | 容器 |
|------|--------|------|
| 隔离级别 | 硬件级（完全隔离） | 进程级（共享内核） |
| 启动时间 | 分钟级 | 秒级（毫秒级） |
| 镜像大小 | GB 级 | MB 级 |
| 性能损耗 | 5-15% | < 5% |
| 密度 | 每主机 10-50 个 | 每主机 100-1000 个 |
| OS 支持 | 任意 OS | 同内核类型 |
| 安全性 | 强隔离 | 较弱（共享内核） |
| 资源占用 | 高（需完整 OS） | 低（共享内核） |

## 4.2 容器的隔离技术

容器通过 Linux 内核特性实现隔离：

### Namespace（命名空间）

| Namespace | 隔离内容 | 说明 |
|-----------|----------|------|
| PID | 进程 ID | 容器内进程从 PID 1 开始 |
| NET | 网络栈 | 独立网络设备、IP、端口 |
| IPC | 进程间通信 | 独立消息队列、信号量 |
| MNT | 文件系统挂载 | 独立文件系统视图 |
| UTS | 主机名和域名 | 独立主机名 |
| USER | 用户和组 | 独立 UID/GID 映射 |

\`\`\`bash
# 查看 Docker 容器的 Namespace
docker inspect --format '{{.State.Pid}}' container_name
ls -la /proc/<PID>/ns
\`\`\`

### Cgroups（控制组）

Cgroups 限制和监控容器资源使用：

| 子系统 | 控制内容 |
|--------|----------|
| cpu | CPU 时间分配 |
| cpuset | CPU 核心绑定 |
| memory | 内存使用限制 |
| blkio | 块设备 I/O 限制 |
| devices | 设备访问控制 |
| net_cls | 网络分类标记 |

\`\`\`bash
# 查看容器的 Cgroup 限制
cat /sys/fs/cgroup/memory/docker/<container_id>/memory.limit_in_bytes
cat /sys/fs/cgroup/cpu/docker/<container_id>/cpu.cfs_quota_us
\`\`\`

### UnionFS（联合文件系统）

容器镜像的分层存储：

\`\`\`
┌─────────────────────┐
│  可写层（Container Layer）│  ← 容器运行时的修改
├─────────────────────┤
│  Layer 3: 应用代码    │  ← COPY 指令
├─────────────────────┤
│  Layer 2: 依赖安装    │  ← RUN pip install
├─────────────────────┤
│  Layer 1: 基础镜像    │  ← FROM python:3.9
└─────────────────────┘
\`\`\`

## 4.3 适用场景对比

### 适合虚拟机的场景

| 场景 | 原因 |
|------|------|
| 运行不同操作系统 | 需要完整的 OS 支持（如 Windows on Linux） |
| 强安全隔离需求 | 多租户环境、合规要求 |
| 内核级操作 | 需要修改内核参数、加载内核模块 |
| 传统应用 | 不支持容器化的遗留应用 |
| 数据库 | 高 I/O、大内存需求 |
| VDI | 虚拟桌面基础设施 |

### 适合容器的场景

| 场景 | 原因 |
|------|------|
| 微服务架构 | 快速部署、弹性伸缩 |
| CI/CD | 一致的构建和运行环境 |
| 开发环境 | 快速启动、易于复制 |
| 无状态应用 | Web 服务、API 服务 |
| 批处理任务 | 短生命周期任务 |
| Dev/Test | 资源高效利用 |

### 混合使用

\`\`\`
生产环境推荐：虚拟机 + 容器混合

物理服务器
├── 虚拟机 1（数据库）     ← 虚拟机：强隔离、高性能 I/O
├── 虚拟机 2（Kubernetes 节点）
│   ├── 容器：Web 服务     ← 容器：快速部署、弹性伸缩
│   ├── 容器：API 服务
│   └── 容器：缓存服务
└── 虚拟机 3（监控/日志）  ← 虚拟机：稳定性要求高
\`\`\`

## 4.4 性能对比

### 启动时间

\`\`\`
虚拟机启动：
  BIOS → GRUB → 内核 → init/systemd → 服务 → 应用
  时间：30s - 5min

容器启动：
  创建 Namespace → 挂载镜像 → 启动进程
  时间：0.05s - 5s
\`\`\`

### 资源开销

\`\`\`
虚拟机额外开销：
  - Guest OS 内核：~200-500MB 内存
  - Guest OS 服务：~100-300MB 内存
  - 磁盘：~2-20GB（完整 OS）
  - CPU 虚拟化损耗：5-15%

容器额外开销：
  - 容器运行时：~10-50MB 内存
  - 磁盘：~5-500MB（分层镜像）
  - CPU 损耗：< 5%
\`\`\`

### I/O 性能

\`\`\`bash
# 虚拟机磁盘 I/O（KVM virtio）
# 接近原生性能的 90-95%

# 容器磁盘 I/O
# 几乎等于原生性能（直接使用宿主机文件系统）

# 网络性能
# 虚拟机：virtio 网卡，接近原生 95%
# 容器：veth pair + bridge，接近原生 98%
# 容器 host 网络模式：等于原生性能
\`\`\`

## 4.5 安全性对比

| 安全维度 | 虚拟机 | 容器 |
|----------|--------|------|
| 隔离强度 | 硬件级（独立内核） | 进程级（共享内核） |
| 逃逸风险 | 低（需突破 Hypervisor） | 较高（内核漏洞影响所有容器） |
| 攻击面 | Guest OS + 应用 | 容器 + 共享内核 |
| 安全加固 | OS 加固 + 防火墙 | 镜像扫描 + 安全策略 |

### 容器安全最佳实践

\`\`\`bash
# 1. 使用最小化基础镜像
FROM alpine:3.18          # 而非 ubuntu:latest

# 2. 不以 root 运行
USER nobody

# 3. 只读文件系统
docker run --read-only ...

# 4. 限制资源
docker run --memory=512m --cpus=1 ...

# 5. 禁用不需要的能力
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE ...

# 6. 使用安全模块
docker run --security-opt=no-new-privileges ...
\`\`\`
