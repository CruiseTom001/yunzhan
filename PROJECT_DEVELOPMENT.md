## 1. 文档定位与必读规则

本文件描述云栈的项目目标、架构边界、核心业务不变量、开发流程和交付标准。它与 `AGENTS.md` 共同构成本仓库的开发基线：

1. 开始任何分析、开发、修改、测试、审查、发布或故障处理前，必须完整阅读 `AGENTS.md`。
2. 随后必须完整阅读本文件 `PROJECT_DEVELOPMENT.md`，确认当前任务涉及的架构、数据和兼容性边界。
3. 修改代码前必须运行 `git status -sb`，保护用户和其他开发者已有的工作树改动。
4. 若任务涉及专项设计文档，还必须继续阅读 `docs/` 下对应的设计、部署或发布说明。

两份基线文档的职责不同：

- `AGENTS.md`：不可绕过的编码、安全、质量和 Git 强制规则。
- `PROJECT_DEVELOPMENT.md`：项目架构、业务不变量、按规模执行的开发流程和文件导航。

发生冲突时，按以下顺序处理：用户对当前任务的明确要求、`AGENTS.md` 强制规则、本文件、最新专项设计文档、现有代码与测试。任何偏离强制规则的行为都必须先说明影响并取得用户明确同意。

## 2. 项目目标与当前技术栈

云栈是面向运维学习的 Web 与桌面一体化平台，核心闭环包括课程学习、模拟 Linux 终端、严格实验判题、测验复习、学习进度、账号与云同步、管理后台和桌面端发布。

当前主要技术栈：

| 层级 | 技术与职责 |
| --- | --- |
| Web 前端 | Vue 3、TypeScript、Pinia、Vue Router、Vite、Tailwind CSS |
| 桌面端 | Electron，复用 Web 构建产物并提供受限 IPC 和本地进度存储 |
| 服务端 | Node.js、Express，提供账号、进度、公告、反馈、审计和版本发布 API |
| 数据库 | PostgreSQL，使用 `server/migrations/` 中的顺序迁移维护结构 |
| 测试 | Vitest，覆盖前端服务模块、终端沙箱、云同步和服务端验证 |
| 部署 | Vercel 承载前端与 Serverless API，PostgreSQL 承载云端业务数据 |

运行时要求以 `package.json` 为准，当前 Node.js 支持范围为 `>=22 <25`。

## 3. 工作区与关键文件地图

### 3.1 前端

- `src/main.ts`：Vue 应用入口。
- `src/App.vue`：全局布局、通用浮层和页面容器。
- `src/router/index.ts`：路由、登录守卫和超级管理员权限边界。
- `src/stores/auth.ts`：认证用户、会话和账号状态。
- `src/stores/progress.ts`：学习进度、本地持久化和云同步协调。
- `src/pages/`：页面级工作流，管理后台页面也位于此目录。
- `src/components/`：可复用 UI、全局导航、AI 与终端组件。
- `src/utils/apiClient.ts`：统一 API 请求基础设施。
- `src/utils/cloudSync.ts`：云端进度同步与版本冲突合并。
- `src/utils/storage.ts`：浏览器和桌面端进度存储适配。
- `src/utils/terminalSandbox.ts`：训练终端命令语义与执行结果。
- `src/utils/markdown.ts`：Markdown 渲染与统一安全清洗。
- `src/data/courses/`、`src/data/markdown/`：课程元数据和章节正文。
- `src/data/labs.ts`：实验步骤与判题条件。
- `src/data/quizzes/`：课程测验数据。

### 3.2 服务端与数据库

- `api/index.mjs`：Vercel Function 入口。
- `server/index.mjs`：Express API 主入口及路由。
- `server/runtime-config.mjs`：运行时配置解析与安全边界。
- `server/validation.mjs`：服务端输入验证工具。
- `server/email-service.mjs`：邮件发送。
- `server/email-verification.mjs`：验证码生成、摘要、频控和验证。
- `server/migrate.mjs`：数据库迁移执行器。
- `server/migrations/*.sql`：按编号顺序执行的数据库迁移。

### 3.3 Electron、构建与文档

- `electron/main.cjs`：安全窗口、导航策略和本地进度 IPC。
- `electron/preload.cjs`：渲染进程可使用的白名单桥接。
- `electron/api-config.json`：桌面端生产 API Origin。
- `scripts/`：源码策略、章节对账、版本同步、图标和辅助脚本。
- `docs/superpowers/specs/`：模块级或高风险功能设计文档。
- `docs/superpowers/plans/`：已确认设计对应的实施计划。
- `docs/ACCOUNT_DEPLOYMENT.md`：账号服务部署说明。
- `docs/DEPLOYMENT_ARCHITECTURE_GUIDE.md`：部署架构说明。
- `RELEASE.md`、`CHANGELOG.md`：桌面发布流程与版本变更记录。

旧的项目介绍或交接文档可能是历史快照。判断当前行为时，以代码、测试、迁移、`package.json` 和最新专项文档为准。

## 4. 按任务规模选择开发流程

### 4.1 咨询、排查和代码审查

适用范围：解释代码、定位故障、评估方案、检查线上或仓库状态。

执行要求：

1. 阅读两份基线文档。
2. 只做必要的只读检查。
3. 说明证据、根因、影响范围和建议。
4. 用户只要求诊断时，不擅自实现修复、提交或推送。

### 4.2 小型修改快速通道

适用范围：局部样式、文案、明确的单点缺陷、小范围配置调整。

无需额外创建设计文档，但必须：

1. 阅读目标文件、直接调用方和相关测试。
2. 确认没有破坏固定导航、移动端布局、主题或可访问性。
3. 保持最小变更，不夹带重构。
4. 完成全部强制质量门禁。

### 4.3 模块级功能

适用范围：新增页面、跨多个前端模块、增加 API、改变状态模型或引入新业务流程。

执行流程：

1. 在 `docs/superpowers/specs/` 创建或更新设计文档，命名建议为 `YYYY-MM-DD-<topic>-design.md`。
2. 设计文档至少包含：目标、非目标、用户流程、数据结构、接口、异常状态、安全边界、兼容性、测试和回滚方案。
3. 对存在重要产品选择或兼容性风险的设计，编码前取得用户确认。
4. 在 `docs/superpowers/plans/` 编写实施计划，命名建议为 `YYYY-MM-DD-<topic>.md`。
5. 按计划逐项实现和验证，完成后同步更新文档状态。

### 4.4 高风险变更

以下变更必须按模块级流程执行，并进行旧数据、桌面升级与回滚评估：

- 学习进度结构、同步合并规则或账号隔离方式。
- 实验判题、命令历史或终端沙箱语义。
- 数据库迁移、账号安全、会话、验证码和权限模型。
- Electron IPC、`appId`、本地 `userData`、更新与卸载行为。
- 生产 API Origin、CORS、Cookie、Vercel、数据库或桌面发布配置。
- 删除、覆盖、导入、清空数据以及批量管理操作。

## 5. 开发前检查清单

每次修改代码前依次完成：

1. 阅读 `AGENTS.md` 和 `PROJECT_DEVELOPMENT.md`。
2. 运行 `git status -sb`，记录已有修改和未跟踪文件。
3. 阅读目标模块、类型、调用方、持久化结构和现有测试。
4. 检查是否存在相关的设计、部署、迁移或发布文档。
5. 识别外部输入、权限、秘密、备份和失败恢复边界。
6. 评估 Web、Electron、旧数据和移动端兼容性。
7. 新增依赖前说明必要性、体积、安全和后续维护成本。

不得因为已有代码“看起来可用”而跳过调用链和测试检查。

## 6. 不可破坏的业务不变量

### 6.1 学习进度与同步

- 学习进度必须保留本地兜底，云端失败不能清空本地数据。
- 云同步冲突必须按明确规则合并并重试，不能简单以后写覆盖先写。
- 本地缓存必须按账号隔离，退出或切换账号不能泄漏其他用户进度。
- 导入、覆盖、清空和删除进度前必须创建可恢复备份。
- 改动进度结构时必须兼容旧数据，必要时提供版本化迁移。

### 6.2 课程、终端与实验

- 终端是训练沙箱；不支持的行为必须明确提示“模拟”或“不支持”。
- 输出语义应尽量与真实 Linux 一致，不得伪装实际系统执行。
- 实验只能根据实验开始后真实执行且退出码成功的命令判定。
- 禁止预写命令历史、模糊包含过关或使用无关输出过关。
- 课程元数据声明的章节数必须与 Markdown 文件数量一致。
- 题目、答案、解释、命令参数和实验步骤必须事实一致。

### 6.3 账号、权限与安全

- 浏览器会话使用安全 Cookie；Token、密码、验证码和秘密不得进入前端持久化或日志。
- 数据库只保存必要的摘要、哈希和业务字段，不保存验证码明文。
- 公开、登录用户和超级管理员 API 必须保持明确权限边界。
- 最后一个可用超级管理员必须受到保护。
- 外部 JSON、请求参数、导入文件和 IPC 参数必须验证类型、长度、枚举和身份范围。
- 管理操作应写入审计日志，但日志不得包含密码、Token、验证码或密钥。

### 6.4 数据库迁移

- 已部署的迁移文件视为不可修改，只能追加新的顺序迁移。
- 迁移应可重复执行或具备明确的幂等保护。
- 结构变化必须考虑旧服务与新服务短时间并存的兼容性。
- 删除字段、收紧约束或批量转换数据前必须有备份和回滚方案。
- 生产迁移完成前，不得让前端默认依赖尚不存在的新结构。

### 6.5 Electron 与桌面升级

- 保持 `contextIsolation: true`、`nodeIntegration: false`、`sandbox: true`。
- preload 只暴露明确白名单能力，IPC 参数必须验证。
- 系统命令使用 `execFile` 和参数数组，不经 shell 拼接执行。
- `appId=com.yunzhan.app` 不得改变。
- 更新和卸载不得默认删除 `userData`，覆盖安装必须保留学习进度。
- `release/`、安装包和构建产物不得提交到 Git。

## 7. 前端与 UI 实现约定

- 页面状态必须覆盖加载、成功、空结果、失败和禁用状态。
- 固定顶部导航高度为 `h-16`；使用全站导航的页面应采用类似 `pt-24 pb-16` 的安全内容间距，避免标题和按钮被遮挡。
- 页面在桌面和移动视口不得横向溢出；操作区在窄屏应换行或降级为图标按钮。
- 表单字段必须有标签，错误信息靠近相关操作，提交时禁用重复操作。
- 图标按钮必须提供 `title` 或可访问名称。
- 重要状态不得只用颜色表达，同时提供文字或图标。
- Vue 组件卸载时必须清理监听器、定时器、Observer 和第三方实例。
- 共享业务状态进入 Pinia 或明确的服务模块，页面组件不重复维护权威状态。
- Markdown 或富文本统一经过 DOMPurify 清洗，组件不得自行拼接未清洗 HTML。

## 8. 服务端与 API 实现约定

- 路由负责鉴权、输入验证和 HTTP 响应，复杂业务拆入具名函数或服务模块。
- 外部数据以不可信数据处理，验证必填字段、类型、长度、格式、枚举和资源归属。
- 异常必须形成闭环：记录安全摘要、返回稳定中文错误，不泄漏堆栈和秘密。
- 新增接口时同步定义前端响应类型守卫，不能用类型断言代替运行时验证。
- 列表接口应设置分页上限；文件、正文和 AI 返回应设置体积上限。
- 删除和高风险写操作必须有权限检查、目标确认、备份或审计。
- 登录、验证码和敏感接口必须保留频控与防枚举设计。

## 9. 测试与质量门禁

测试文件遵循现有邻近放置方式：前端与工具测试使用 `*.test.ts`，服务端测试使用 `*.test.mjs`。

修改完成后必须依次运行：

1. `npm run policy:check`
2. `npm run check`
3. `npm run lint`
4. `npm run server:check`
5. `npm test`
6. `npm run build`
7. `git diff --check`

修改依赖时额外运行 `npm audit --omit=dev`。涉及数据库时应在可用的测试数据库执行迁移和接口验证；涉及 Electron 时应验证 IPC、安全配置、本地进度和覆盖安装影响。

任何必需检查失败时，不得宣布完成、提交、推送、部署或生成正式安装包。不得用 `build:fast` 代替正式构建门禁。

## 10. 版本、发布与部署

项目遵循语义化版本：

- 主版本：不兼容的数据或架构变更。
- 次版本：向后兼容的新功能。
- 修订版本：缺陷修复、样式和小范围内容调整。

发布要求：

1. 版本号以 `package.json` 为主，并通过版本同步检查。
2. 用户可见变化同步写入 `CHANGELOG.md`。
3. 桌面发布前阅读 `RELEASE.md` 并运行 `npm run release:windows`。
4. 安装包发布到明确的下载位置，但不提交 `release/`。
5. 桌面版本记录中的版本号、最低兼容版本、下载地址和发布说明必须真实一致。
6. 推送、部署、数据库生产迁移、创建 Release 和修改线上数据都需要用户明确授权。
7. 上线后验证健康检查、关键业务入口和回滚条件，不能只依据构建成功判断上线成功。

## 11. Git 与协作规则

- 保留工作树已有修改；不得擅自回滚、覆盖、暂存或提交其他人的文件。
- 一个提交只表达一个清晰目的，不夹带生成物和无关格式化。
- 提交前检查 `git diff --stat`、`git diff` 和 `git status -sb`。
- 未经明确要求，不创建提交、不推送、不创建 PR、不修改线上数据。
- 分支、提交和 PR 标题使用可读、可追踪的描述。
- 发现规范、设计和代码不一致时，先确认真实行为，再同步更新文档。

## 12. 完成定义与交付说明

一项代码任务只有同时满足以下条件才算完成：

- 请求的行为已经实现，非占位或伪流程。
- 加载、成功、空结果和失败路径得到处理。
- 安全、权限、备份、兼容性和回滚边界已评估。
- 相关测试已新增或更新。
- 全部强制门禁通过。
- 变更清单中没有无关文件或生成物。
- 最终说明包含行为变化、验证命令、未执行检查、兼容性影响和遗留风险。

如果仅完成本地修改，应明确说明“尚未提交/推送/上线”；如果完成线上变更，应给出可验证的版本、提交或地址。

## 13. 文档维护

出现以下变化时，应在同一个任务中更新本文件：

- 技术栈、目录职责或关键入口变化。
- 新增不可破坏的业务不变量。
- 测试、构建、发布或部署流程变化。
- 新增长期使用的设计文档目录或协作流程。

维护时只写稳定且可复用的项目事实，短期任务进度应进入专项设计、实施计划、交接文档或 Git 提交记录，不应堆积在本文件中。
